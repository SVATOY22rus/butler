Идея проекта
Нужно сделать веб-панель для Linux-сервера, на котором разные сервисы работают 
на своих портах. Сервисы остаются жить на отдельных портах, а доступ к ним 
управляется централизованно через собственную админку. Для хранения правил и комментариев подходит SQLite, для веб-интерфейса — Flask, а реальное разрешение
или блокировка доступа будет применяться через nftables sets.

Пример желаемого сценария:

rто-то пытается подключиться к сервису на порту 8011;

система видит эту попытку;

в панели видно IP-адрес, порт, время и количество попыток;

администратор узнаёт адрес, пишет комментарий вроде "клиент Иванов";

нажимает кнопку "Разрешить";

IP попадает в белый список нужного сервиса;

клиент повторяет подключение и получает доступ.

Что именно хотим получить
Финальный продукт — это локальная веб-морда для администратора. Она должна 
показывать попытки подключений, разрешённые адреса, запрещённые адреса, список сервисов и журнал изменений. Доступ к сервисам должен зависеть не только от 
самого IP, но и от того, к какому сервису этот IP допущен.

Основные возможности
Просмотр попыток подключения по IP и по сервису.

Добавление IP в белый список.

Добавление IP в чёрный список.

Хранение комментария к IP, например "клиент", "офис", "временный доступ".

Отдельная страница управления белыми списками.

Отдельная страница управления чёрными списками.

Отдельная страница управления сервисами и их портами.

Журнал действий администратора: кто, что и когда изменил.

В дальнейшем — синхронизация с nftables без ручного редактирования правил.

Главная мысль архитектуры
Очень важно понять одну идею: Flask-приложение не будет само фильтровать
сетевой трафик. Оно будет только хранить данные, показывать интерфейс и 
отдавать команды на применение правил. Сам трафик будет фильтровать nftables,
потому что именно firewall принимает решение, пропускать соединение или нет.

Иными словами:

Flask — это панель управления;

SQLite — это память проекта;

nftables — это исполнитель правил;

логи firewall или nginx — это источник событий о попытках подключений.

Почему выбран именно такой стек
Flask
Flask подходит, потому что он проще, чем FastAPI. У него понятные маршруты,
простые HTML-шаблоны.

SQLite
SQLite хорош для MVP, потому что не требует отдельного сервера базы данных. 
Для начала достаточно одного файла базы, который приложение открывает локально.
Для небольшого внутреннего инструмента это очень удобный и понятный вариант.

nftables
nftables — современный стандартный firewall в Linux. Он поддерживает sets,
а sets удобны тем, что можно хранить списки адресов и быстро добавлять или
удалять IP без полной переделки всех правил.

Как мы видим проект по слоям
Чтобы проект было легче понимать, полезно разделить его на 4 слоя.

1. Интерфейс
Это страницы, которые открываются в браузере. На них администратор смотрит
попытки, добавляет IP в списки, редактирует комментарии и управляет сервисами.

2. Логика приложения
Это код Flask, который обрабатывает кнопки и формы. Например, когда
пользователь нажимает кнопку "Разрешить", Flask получает данные формы,
сохраняет запись в БД и запускает синхронизацию firewall.

3. База данных
Это SQLite. В ней лежит всё, что важно сохранить между перезапусками:

сервисы;

IP-адреса;

белые списки;

чёрные списки;

попытки входа;

журнал действий.

4. Сетевой уровень
Это nftables. Он не знает ничего о красивом интерфейсе, клиентах и 
комментариях. Он просто знает: этот IP можно пускать, а этот нельзя.

На этой странице должны быть кнопки:

3. Белый список
Здесь хранятся все разрешённые адреса. Для каждой записи полезно видеть:

IP-адрес;

комментарий;

владелец или клиент;

4. Чёрный список
Здесь хранятся все заблокированные адреса. Поля похожие, но дополнительно полезны:

причина блокировки;

срок окончания блокировки, если она временная;

источник блокировки: вручную или автоматически.

5. Сервисы
Здесь будут описаны сами сервисы:

имя;

порт;

протокол;

описание.

Пример:

CRM — 8011/tcp;

API — 8080/tcp;

Admin Panel — 9000/tcp.

6. Журнал действий
Это страница для спокойствия администратора. На ней видно, кто и когда что
сделал:

добавил IP;

удалил IP;

разрешил доступ к сервису;

заблокировал адрес;

изменил комментарий.

Как будет работать логика разрешения и блокировки
Ниже важная логика простыми словами.

Когда IP пытается подключиться
Клиент стучится на порт сервиса.

Если IP неизвестен или не разрешён, firewall не пускает его и пишет событие
в лог.

Это событие попадает в таблицу attempts.

На странице попыток администратор видит новую запись.

Когда администратор нажимает "Разрешить"
Flask получает IP и выбранный сервис.

Записывает разрешение в БД.

Добавляет комментарий, если он введён.

Пишет запись в audit_log.

Вызывает модуль синхронизации.

Модуль синхронизации добавляет IP в нужный nftables set.

Когда администратор нажимает "Заблокировать"
Flask записывает IP в чёрный список.

Пишет причину и комментарий.

Пишет запись в audit_log.

Модуль синхронизации добавляет IP в blacklist set в nftables.

Как мы будем использовать nftables
Ключевая идея — не писать гигантские правила вручную каждый раз. Вместо этого
будут sets, то есть именованные наборы адресов. nftables умеет использовать
sets в правилах доступа, и именно поэтому такая схема хорошо подходит для динамического окружения.

Примеры наборов:

global_whitelist

global_blacklist

svc_8011_allow

svc_8011_deny

svc_8080_allow

Пример мысли, а не готовое правило:

если IP в глобальном чёрном списке — блокировать сразу;

если IP в deny-списке сервиса — блокировать;

если IP в allow-списке сервиса — разрешить;

иначе залогировать попытку и отклонить соединение.

Почему комментарии храним в БД, а не в firewall
Это очень важное архитектурное решение. Firewall должен быть быстрым и простым. Бизнес-данные вроде "это клиент Петров" или "временный доступ до пятницы"
удобнее хранить в SQLite, потому что база данных лучше подходит для поиска, редактирования, истории и отображения в интерфейсе.

То есть firewall отвечает на вопрос "пускать или не пускать",
а БД отвечает на вопрос "почему это правило существует и кому оно принадлежит".

Какой будет план разработки
Чтобы не запутаться, проект делим на небольшие этапы.

Этап 1. Каркас проекта
Нужно сделать:

структуру каталогов;

Flask-приложение;

базовый шаблон сайта;

подключение SQLite;

создание таблиц.

Результат этапа: приложение запускается и открывается в браузере.

Этап 2. Управление сервисами
Нужно сделать страницу сервисов:

добавление;

редактирование;

удаление;

проверка уникальности порта.

Результат этапа: можно завести список реальных служб сервера.

Этап 3. Белый и чёрный списки
Нужно сделать две отдельные страницы:

список разрешённых IP;

список запрещённых IP.

Результат этапа: можно вручную вести учёт адресов и комментариев.

Этап 4. Попытки подключений
Сначала можно добавить демо-данные или тестовый импорт. Потом подключить
реальные логи firewall или nginx.

Результат этапа: в панели видны реальные события доступа.

Этап 5. Синхронизация с nftables
Нужно сделать Python-модуль, который по данным БД обновляет nftables sets.
Для начала лучше применять изменения через отдельные команды или подготовленный ruleset, а не усложнять сразу всё подряд.

Результат этапа: кнопки в панели реально начинают влиять на доступ.

Этап 6. Полировка
Нужно добавить:

авторизацию в админке;

фильтры и поиск;

временные разрешения и блокировки;

страницу журнала действий;

более удобный UI.

Как запускать проект позже
Ниже базовые шаги, чтобы спустя несколько дней быстро вернуться к проекту.

1. Активировать виртуальное окружение
Пример для Linux:

bash
source .venv/bin/activate
2. Установить зависимости
bash
pip install -r requirements.txt
3. Запустить приложение
bash
python run.py
4. Открыть в браузере
Обычно Flask запускается локально на адресе http://127.0.0.1:5000.
